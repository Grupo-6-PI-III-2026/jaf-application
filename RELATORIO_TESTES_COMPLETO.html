Bug #2: FuncionarioService - Validação de Email
ALTA

Arquivo: FuncionarioService.java (método atualizar)

Problema:

O método atualizar não verificava se o novo email já pertencia a outro usuário, permitindo duplicidade de emails no sistema.

Impacto:

Dois usuários poderiam ter o mesmo email após atualização, quebrando a unicidade e causando problemas de autenticação.

Correção Aplicada:

public FuncionarioResponseDto atualizar(Long id, FuncionarioDto dto) { Funcionario existente = funcionarioRepository.findById(id) .orElseThrow(() -> new NotFoundException("Usuário não encontrado.")); // Validar email único se estiver sendo alterado if (!existente.getEmail().equals(dto.getEmail())) { funcionarioRepository.findByEmailIgnoreCase(dto.getEmail()) .ifPresent(f -> { throw new Conflict("E-mail já cadastrado para outro usuário."); }); } existente.setNome(dto.getNome()); existente.setEmail(dto.getEmail()); // ... resto do código }
Status: ✅ CORRIGIDO
Melhoria #3: Validação de Data Range em Obra
MÉDIA

Arquivo: ObraDto.java

Problema:

Não havia validação para garantir que a data de término seja posterior à data de início da obra.

Impacto:

Obras poderiam ter datas de término anteriores às datas de início, causando inconsistências nos dados.

Correção Aplicada:

  • Criado anotação customizada @DateRange
  • Criado validator DateRangeValidator
  • Adicionado validação em ObraDto
@DateRange(startDateField = "dtInicio", endDateField = "dtTerminoPrevisto") public class ObraDto { // ... campos }
Status: ✅ CORRIGIDO

🛡️ Problemas de Segurança Identificados

6 Issues
Issue #1: CORS Configuration Inseguro
ALTA

Arquivo: SecurityConfiguracao.java (linha 122)

Problema:

configuracao.setAllowedOriginPatterns(List.of("*"));

Impacto:

Permite CORS de qualquer origem, o que é inseguro em produção e pode levar a ataques CSRF.

Recomendação:

// Especificar apenas origens permitidas configuracao.setAllowedOrigins(Arrays.of(origins)); configuracao.setAllowedOriginPatterns(null); // Remover wildcard
Status: ⚠️ NÃO CORRIGIDO (Requer Ação)
Issue #2: Falta de Validação de Complexidade de Senha
MÉDIA

Problema:

Não há validação de complexidade de senha (mínimo de caracteres, maiúsculas, números, etc.).

Recomendação:

Adicionar validação no DTO e/ou no PasswordEncoder customizado com requisitos mínimos.

Status: ⚠️ NÃO CORRIGIDO (Requer Ação)
Issue #3: Token JWT Sem Validação de Revogação
MÉDIA

Problema:

Não há mecanismo para revogar tokens (logout, troca de senha, etc.). Tokens permanecem válidos até expiração.

Recomendação:

Implementar blacklist de tokens ou usar tokens de curta duração com refresh tokens.

Status: ⚠️ NÃO CORRIGIDO (Requer Ação)
Issue #4: Exposição de Informações Sensíveis em Exceções
MÉDIA

Problema:

Mensagens de exceção podem expor informações internas do sistema para usuários mal-intencionados.

Recomendação:

Implementar um GlobalExceptionHandler para padronizar e sanitizar mensagens de erro.

Status: ⚠️ NÃO CORRIGIDO (Requer Ação)
Issue #5: Falta de Rate Limiting
MÉDIA

Problema:

Não há limitação de taxa de requisições, deixando o sistema vulnerável a ataques de DoS e força bruta.

Recomendação:

Implementar rate limiting com Spring Boot Starter ou bucket4j.

Status: ⚠️ NÃO CORRIGIDO (Requer Ação)
Issue #6: Ausência de Auditoria
BAIXA

Problema:

Não há logs de auditoria para ações críticas (criação, deleção, atualização de dados).

Recomendação:

Implementar sistema de auditoria com @AuditLogging ou Spring Data Envers.

Status: ⚠️ NÃO CORRIGIDO (Requer Ação)

📋 Regras de Negócio Identificadas

Validação

Regras de Funcionários

1
Funcionários Internos:
• Devem ter senha obrigatória
• Podem ter documento opcional
• PRECISAM estar alocados para ter gastos registrados
• Email deve ser único
2
Funcionários Externos:
• NÃO têm senha (não acessam o sistema)
• Documento é obrigatório
• Email é opcional
• NÃO precisam estar alocados para ter gastos
• Podem ter gastos em múltiplas obras simultaneamente
3
Atualização de Funcionário:
• Email só pode ser alterado se não existir em outro usuário
• Tipo de funcionário pode ser alterado

Regras de Obras

1
Validações:
• Título deve ser único
• Data de término deve ser posterior à data de início
• Data de término não pode estar no passado
2
Escopo de Visualização:
• ADMIN: vê todas as obras
• GESTOR_OBRA: vê apenas obras onde está alocado
• OPERADOR_LANCAMENTO: vê apenas obras onde está alocado

Regras de Alocações

1
Regras:
• Um funcionário não pode ser alocado duas vezes na mesma obra
• Funcionários externos não podem visualizar alocações
2
Escopo de Visualização:
• ADMIN: vê todas as alocações
• GESTOR_OBRA: vê apenas suas alocações
• OPERADOR_LANCAMENTO: visualização limitada
• EXTERNO: não visualiza alocações

Regras de Gastos

1
Regras de Alocação:
• Funcionários internos: PRECISAM estar alocados na obra para ter gastos
• Funcionários externos: NÃO precisam estar alocados
2
Escopo de Visualização:
• ADMIN: vê todos os gastos
• GESTOR_OBRA: vê gastos das obras onde está alocado
• OPERADOR_LANCAMENTO: visualização limitada
• EXTERNO: não visualiza gastos

Regras de Presença

1
Regras:
• Funcionário deve estar alocado na obra
• Não pode haver duplicidade (mesmo funcionário, obra, data)
• Horário de entrada deve ser anterior ao horário de saída
• Data não pode estar no futuro
